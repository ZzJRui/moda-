# AI Closet Stylist MVP PRD v2

版本：v0.2  
日期：2026-07-04  
产品形态：手机优先的响应式 Web  
开发优先级：先主开发手机竖屏页面，再适配宽屏桌面页面

## 1. 版本结论

第 2 版不先做原生 App，而是先做一个手机优先的响应式 Web 版 AI 数字衣柜。

用户用手机浏览器打开网址，就能完成上传衣物、管理数字衣柜、通过 AI 获得搭配推荐、查看动画结果，并保存穿搭截图。桌面端也可以访问，但第一版只做必要适配，不作为主要体验。

后续如果要做 App，可以再用 React Native + Expo 单独开发移动端壳层，继续复用同一套后端 API、数据库结构和 AI 推荐服务。

## 2. 一句话说明

做一个手机优先的 Web 版 AI 数字衣柜 MVP：用户用手机上传单品，系统生成数字衣柜；用户通过 AI 对话获得基于已有衣物的搭配推荐，前端用动画展示结果，并允许用户保存穿搭截图。

## 3. 为什么先做响应式 Web

现阶段最重要的是验证产品闭环，而不是先进入 App 打包、上架和原生能力适配。

响应式 Web 的优势：

- 用户可以直接通过网址访问，不需要安装。
- 开发和调试更快，适合 MVP。
- 手机端可以做得接近 App：底部导航、大按钮、拍照或相册上传、滑动浏览。
- 后端、AI 推荐和数据结构可以长期复用。
- 以后迁移到 React Native + Expo 时，不需要推翻产品逻辑。

本版本的取舍：

- 先保证手机浏览器体验。
- 桌面端只做合理适配。
- 不做 App Store 或安卓应用市场上架。
- 不把 AI 模型放到手机本地运行。

## 4. 目标用户与使用场景

目标用户是希望快速整理衣柜、获得日常穿搭建议的个人用户。第一版按单用户本地演示或轻量在线演示设计。

典型场景：

- 用户用手机拍照或从相册选择衣物图片。
- 系统把衣物保存为数字衣柜单品。
- 用户出门前输入“今天想去公园玩，帮我搭配一套舒服一点的”。
- 系统从已有衣物中推荐上衣、下装和鞋子。
- 页面用老虎机式动画展示 AI 正在搭配的过程。
- 用户保存满意的搭配截图，在“我的喜欢”里查看。

## 7. 响应式策略

### 7.1 总体原则

采用 Mobile First 写法。默认样式按手机竖屏设计，再通过断点适配平板和桌面。

建议断点：

```text
默认：手机竖屏
>= 768px：平板和小宽屏
>= 1024px：桌面宽屏
```

### 7.2 手机端导航

手机竖屏使用底部 Tab 导航。

```text
[ 页面内容 ]

[ 底部导航 ]
衣柜 | AI | 搭配 | 喜欢
```

导航顺序：

1. 衣柜
2. AI
3. 搭配
4. 喜欢

原因：

- 衣柜是资产入口。
- AI 是核心卖点。
- 搭配是手动探索。
- 喜欢是结果沉淀。

手机端要求：

- 底部导航固定在屏幕底部。
- 每个点击区域不小于 44px。
- 页面内容预留底部安全距离，避免被导航遮挡。
- iPhone 需要处理 `safe-area-inset-bottom`。
- 不依赖 hover 操作。

### 7.3 桌面端导航

宽屏桌面使用顶部导航栏。

```text
[ AI Closet ]  衣柜  AI 搭配  手动搭配  我的喜欢

[ 页面内容 ]
```

桌面端要求：

- 顶部导航横向排列。
- 内容区域居中，并设置最大宽度。
- 衣物列表可以使用多列网格。
- AI 搭配页可以做左右分栏：左侧对话，右侧推荐结果。

### 7.4 导航显示规则

同一时间只显示一种导航。

```text
手机端：显示底部导航，隐藏顶部导航
桌面端：显示顶部导航，隐藏底部导航
```

## 8. 页面需求

### 8.1 衣柜页 `/closet`

手机端优先级最高。

核心内容：

- 上传按钮。
- 筛选标签。
- 衣物卡片列表。
- 空状态提示。

手机端布局：

```text
页面标题
上传衣物按钮
标签筛选
衣物卡片网格
```

衣物卡片信息：

- 单品图片。
- 品类。
- 颜色。
- 风格。

第一版支持品类：

- 上衣：`top`
- 下装：`bottom`
- 鞋子：`shoes`

第一版支持标签：

- 白色
- 黑色
- 蓝色
- 休闲
- 甜酷
- 通勤
- 运动

上传流程：

```text
选择图片或拍照
-> 上传到后端
-> 保存原图
-> 生成白底处理图
-> mock 或规则生成标签
-> 写入数据库
-> 返回单品数据
-> 页面刷新列表
```

必要状态：

- 上传中。
- 上传失败。
- 图片格式不支持。
- 衣柜为空。
- 筛选无结果。

### 8.2 AI 搭配页 `/chat`

页面目标：让用户用一句话获得穿搭推荐。

手机端布局：

```text
对话区域
推荐动画区域
推荐结果
推荐理由
保存按钮
输入框
```

用户输入示例：

```text
今天想去公园玩，帮我搭配一套舒服一点的。
```

推荐输出：

```json
{
  "top_id": 12,
  "bottom_id": 35,
  "shoes_id": 48,
  "reason": "白色上衣和浅蓝色下装比较清爽，运动鞋适合走路，整体适合公园休闲场景。"
}
```

推荐动画：

- 三行分别代表上衣、下装、鞋子。
- 每行横向滚动，模拟老虎机效果。
- 动画结束后停在推荐单品。
- 动画时间控制在 1.5 到 3 秒。
- 用户开启减少动态效果时，可以缩短或关闭动画。

兜底规则：

- 如果某个品类没有单品，提示用户先上传该品类。
- 如果 AI 或规则无法匹配，随机选择可用组合。
- 推荐结果必须引用真实存在的单品 ID。

### 8.3 手动搭配页 `/style`

页面目标：用户不问 AI，也能自己浏览和组合衣物。

手机端布局：

```text
当前穿搭预览
上衣横向滑动区
下装横向滑动区
鞋子横向滑动区
保存按钮
```

核心能力：

- 每个品类可以左右滑动。
- 当前选中的三件单品组成一套搭配。
- 用户可以保存当前组合截图。
- 某个品类为空时提示补充单品。

### 8.4 我的喜欢页 `/favorites`

页面目标：查看保存过的穿搭截图。

手机端布局：

```text
收藏截图列表
保存时间
推荐理由
```

第一版不做复杂分类。先支持：

- 保存。
- 查看。
- 空状态。
- 图片加载失败提示。

## 9. AI 策略

第一版不直接依赖真实模型。先用 mock 和规则跑通闭环。

### 9.1 打标签策略

第一版可以使用三种方式之一：

- 上传后用户手动选择品类、颜色和风格。
- 根据文件名或简单规则猜测。
- 使用 mock 数据返回固定标签。

接口返回格式保持稳定：

```json
{
  "category": "top",
  "color": "white",
  "style": "casual"
}
```

后续接入真实模型时，只替换标签服务内部实现，前端和数据库不需要大改。

### 9.2 推荐策略

第一版推荐逻辑：

1. 读取衣柜里的全部单品。
2. 根据用户输入提取关键词。
3. 优先匹配颜色和风格。
4. 每个品类选择一个单品。
5. 返回推荐理由。

示例规则：

- 用户输入“白色”，优先选择白色单品。
- 用户输入“公园”“舒服”“休闲”，优先选择休闲或运动风格。
- 用户输入“通勤”，优先选择通勤风格。
- 缺少匹配项时随机补齐。

接口返回格式保持稳定：

```json
{
  "top_id": 1,
  "bottom_id": 3,
  "shoes_id": 8,
  "reason": "这套搭配偏休闲，颜色清爽，适合日常出门。"
}
```

后续接入真实 AI 后，仍然要求模型只返回已有单品 ID，不能生成不存在的衣服。

## 13. 移动端体验要求

手机端是第一优先级。

要求：

- 页面宽度从 360px 起可用。
- 核心操作区域适合单手点击。
- 按钮和 Tab 点击区域不小于 44px。
- 上传入口明显。
- 底部导航不遮挡保存按钮。
- 图片列表加载时要有占位状态。
- 动画区域高度受控，不挤压输入框和保存按钮。
- 所有空状态都要告诉用户下一步做什么。

## 14. 桌面端适配要求

桌面端不是第一优先级，但不能不可用。

要求：

- 顶部导航替代底部导航。
- 内容区设置最大宽度。
- 衣柜页使用更多列展示卡片。
- AI 搭配页可以改为左右分栏。
- 底部 Tab 在桌面端不显示。

## 15. 验收标准

### 15.1 手机端

- 手机浏览器能打开并完成主要流程。
- 底部导航可切换四个页面。
- 用户能上传衣物图片。
- 上传后能在衣柜中看到单品。

### 15.2 桌面端

- 桌面浏览器能打开所有页面。
- 顶部导航可切换四个页面。
- 页面布局不挤压、不重叠。
- 衣柜和收藏列表在宽屏下展示正常。

### 15.3 AI 与数据

- 推荐结果只引用已有单品 ID。
- 缺少品类时有明确提示。

## 16. 开发顺序

### 阶段一：手机端静态页面

- 搭建 React + Vite + TypeScript。
- 配置 Tailwind CSS。
- 完成底部 Tab 导航。

- 实现图片上传。
- 保存原图。

### 阶段五：mock / 规则 AI

- 实现规则打标签。

### 阶段六：动画与截图保存

- 实现推荐动画。
- 接入 `html2canvas`。
- 上传截图到后端。

## 18. 当前版本核心判断

第 2 版的关键不是先做一个“能装到手机里的 App”，而是先做一个“手机上真的好用的 Web 产品”。

这样可以更快验证：

- 用户愿不愿意上传衣物。
- AI 推荐是否有吸引力。
- 动画展示是否能让结果更有感觉。
- 保存穿搭截图是否有实际价值。

等这些被验证后，再做 App，会更稳。
