# AI Closet Stylist MVP PRD v3

版本：v0.3  
日期：2026-07-04  
产品形态：手机优先的响应式 Web  
核心变化：底部四页导航 + 中心上传按钮 + AI/手动老虎机搭配 + 喜欢页双列截图流

## 1. 版本结论

第 3 版继续采用手机优先的响应式 Web，不先做原生 App。产品结构从“普通四页面”调整为更接近移动 App 的主框架：

```text
社区 | 我的喜欢 | +上传 | AI 搭配 | 我的衣柜
```

其中 `+上传` 是底部中间主按钮，不算独立页面。点击后支持拍照或从图库选择图片。

第 3 版的产品重点是 AI 搭配页。用户可以通过文字或语音告诉 AI 想穿什么，页面上的三组衣柜老虎机自动转动，最终停在 AI 推荐的上衣、下装和鞋子。用户也可以手动滑动三组衣物，自己搭配。搭配完成后点击保存，系统生成穿搭截图，并在“我的喜欢”页用双列图片流展示。

## 2. 一句话说明

做一个手机优先的 AI 数字衣柜 Web App：用户通过中心按钮拍照或上传单品；AI 搭配页用三组老虎机展示上衣、下装和鞋子，支持 AI 文字/语音推荐和手动滑动选择；保存后的穿搭截图进入“我的喜欢”，以类似小红书的双列图片流展示；“我的衣柜”用于管理和搜索所有单品。

## 3. 产品目标

第一版要验证四件事：

- 用户愿不愿意上传真实衣物。
- 三组老虎机式衣柜是否足够直观、有记忆点。
- AI 对话推荐能否让用户快速得到一套搭配。
- 用户是否愿意保存穿搭截图，并回到“我的喜欢”查看。

本期不追求复杂社区、完整用户体系或真实大模型能力。先用 mock 和规则跑通闭环，后续再替换真实模型。

## 4. 信息架构

### 4.1 底部导航

手机端底部导航固定为四个页面加一个中心操作按钮：

```text
社区     喜欢     +     搭配     衣柜
```

说明：

- 社区：路由固定为 `/community`，第一版使用写死内容。
- 喜欢：路由为 `/favorites`，展示已保存穿搭截图。
- `+`：中心上传按钮，不进入页面，点击后弹出上传面板。
- 搭配：路由为 `/style`，承载 AI 推荐和手动搭配。
- 衣柜：路由为 `/closet`，展示、搜索和管理单品。

建议默认首页进入 `/style`。它最能体现产品差异：一打开就能看到三组衣柜老虎机和 AI 搭配入口。

### 4.2 桌面端导航

桌面端使用顶部导航，但信息架构不变：

```text
AI Closet | 社区 | 我的喜欢 | AI 搭配 | 我的衣柜 | 上传
```

桌面端不是第一优先级，只保证可用、布局不重叠、页面内容能正常访问。

## 5. 页面清单

| 路由 | 页面 | 本期定位 |
| --- | --- | --- |
| `/community` | 社区页 | 写死内容，展示穿搭社区氛围 |

## 6. 中心上传按钮

### 6.1 入口位置

手机端底部导航中间放置一个突出的 `+` 按钮。

```text
社区     喜欢     +     搭配     衣柜
```

点击后弹出上传操作面板。

### 6.2 上传方式

第一版支持：

- 拍照上传。
- 从图库选择。

Web 端可以使用文件选择能力实现：

```text
拍照：input capture="environment"
图库：input type="file"
```

实际兼容性按浏览器能力处理。第一版不要求做到原生 App 级相机体验。

### 6.3 上传流程

```text
点击 +
-> 选择拍照或图库
-> 选择图片
-> 上传后端
-> 保存原图
-> 生成处理图
-> 生成或确认标签
-> 写入衣柜
-> 上传成功提示
-> 新单品出现在我的衣柜和搭配页老虎机中
```

### 6.4 上传后的标签

第一版可以先用 mock 或规则生成标签，也可以让用户手动修正：

```json
{
  "category": "top",
  "color": "white",
  "style": "casual"
}
```

必须保证品类落到三类之一：

- `top`
- `bottom`
- `shoes`

## 7. AI 搭配页 `/style`

### 7.1 页面定位

这是产品核心页。它同时承担：

- AI 推荐。
- 三件单品老虎机展示。
- 手动滑动搭配。
- 保存穿搭截图。

不再单独拆一个“AI 对话页”和一个“手动搭配页”。两种能力集中在同一个搭配页里。

### 7.2 手机端布局

```text
页面标题 / 当前状态

上衣老虎机
下装老虎机
鞋子老虎机

当前搭配预览 / 推荐理由
保存按钮

右下角 AI 对话按钮
底部导航
```

### 7.3 三组老虎机

三组衣物分别对应：

- 上衣。
- 下装。
- 鞋子。

每组展示同品类衣物，支持两种状态：

1. AI 自动转动。
2. 用户手动滑动。

AI 推荐时：

```text
用户输入想法
-> 后端返回 top_id / bottom_id / shoes_id
-> 三组衣物开始横向转动
-> 动画依次停止
-> 停在推荐的三件单品上
-> 显示推荐理由
```

手动搭配时：

```text
用户左右滑动上衣
用户左右滑动下装
用户左右滑动鞋子
当前选中项组成一套搭配
点击保存
```

### 7.4 AI 对话入口

AI 对话按钮固定在页面右下角，位于底部导航上方，避免被遮挡。

按钮点击后弹出对话面板，支持：

- 文字输入。
- 音频输入入口。

第一版音频能力可以先做入口和状态，不强依赖真实语音识别。更稳的方式是：

```text
第一版：文字输入为主，音频按钮可先做 UI 占位或接浏览器语音输入能力
后续：接入真实语音转文字服务
```

用户输入示例：

```text
今天想去公园，帮我搭一套舒服一点的。
```

```text
我想要甜酷一点，适合晚上出门。
```

### 7.5 AI 推荐返回

推荐服务只允许从用户已有衣柜中选择单品。

```json
{
  "top_id": 12,
  "bottom_id": 35,
  "shoes_id": 48,
  "reason": "白色上衣和浅蓝色下装比较清爽，运动鞋适合走路，整体适合公园休闲场景。"
}
```

如果某个品类为空，不能返回无效结果。页面提示：

```text
还缺少鞋子，先上传一双鞋再让 AI 搭配吧。
```

### 7.6 保存穿搭

用户在 AI 推荐结果或手动搭配结果上点击保存后，系统保存两类数据：

1. 穿搭组合数据。
2. 当前穿搭截图。

保存内容示例：

```json
{
  "top_id": 12,
  "bottom_id": 35,
  "shoes_id": 48,
  "source": "ai",
  "prompt": "今天想去公园，帮我搭一套舒服一点的。",
  "reason": "白色上衣和浅蓝色下装比较清爽，运动鞋适合走路。",
  "screenshot_path": "/uploads/favorites/outfit_001.png"
}
```

手动搭配保存时：

```json
{
  "top_id": 12,
  "bottom_id": 35,
  "shoes_id": 48,
  "source": "manual",
  "prompt": null,
  "reason": "手动搭配",
  "screenshot_path": "/uploads/favorites/outfit_002.png"
}
```

## 8. 我的喜欢页 `/favorites`

### 8.1 页面定位

我的喜欢页第一版就是用户的穿搭灵感相册。

第一版只展示保存过的穿搭截图，采用类似小红书的双列图片流。不要一开始展示太多文字和复杂筛选。

### 8.2 手机端布局

```text
页面标题：我的喜欢

[截图卡片] [截图卡片]
[截图卡片] [截图卡片]
[截图卡片] [截图卡片]

底部导航
```

卡片第一版可以只展示：

- 穿搭截图。
- 可选：保存时间或来源标签。

### 8.3 点击行为

点击截图后进入详情页：

```text
/favorites/{id}
```

详情页第一版展示：

- 大图截图。
- 保存时间。
- 来源：AI 推荐 / 手动搭配。
- 返回按钮。

如果数据已经具备，也可以展示：

- 上衣。
- 下装。
- 鞋子。
- AI 推荐理由。

后续可扩展：

- 再次穿这套。
- 重新编辑。
- 删除收藏。
- 分享到社区。

### 8.4 空状态

没有保存内容时显示：

```text
还没有喜欢的穿搭。
去搭配页保存第一套吧。
```

按钮跳转到 `/style`。

## 9. 我的衣柜页 `/closet`

### 9.1 页面定位

我的衣柜页专门管理单品，不承担穿搭组合功能。

它解决三个问题：

- 我上传了哪些衣服。
- 我怎么找到某件衣服。
- 我怎么修正或管理单品信息。

### 9.2 手机端布局

```text
页面标题：我的衣柜
搜索框
品类筛选
单品图片网格
底部导航
```

### 9.3 搜索能力

用户可以搜索：

- 单品名称。
- 品类。
- 颜色。
- 风格。

示例：

```text
白色
运动鞋
黑色
裙子
通勤
```

搜索结果展示对应单品图片。

### 9.4 单品卡片

单品卡片展示：

- 单品图片。
- 名称。
- 品类。
- 颜色。
- 风格。

点击单品后进入单品详情或打开编辑面板。第一版可以先支持查看，后续再支持编辑标签、删除、重新识别。

## 13. AI 和语音策略

### 13.1 第一版 AI

第一版先用 mock 和规则推荐，不强依赖真实模型。

推荐输入：

```json
{
  "text": "今天想去公园，帮我搭一套舒服一点的。"
}
```

推荐输出：

```json
{
  "top_id": 12,
  "bottom_id": 35,
  "shoes_id": 48,
  "reason": "这套偏休闲，颜色清爽，适合公园散步。"
}
```

后续替换真实模型时，保持输入输出结构不变。

### 13.2 语音输入

语音输入第一版可以分两步做：

1. UI 上提供麦克风按钮和录音状态。
2. 能力上先以文字输入为主，语音转文字后续接入。

如果浏览器能力允许，可以先尝试 Web Speech API；如果兼容性不稳定，就作为后续功能。

## 15. 响应式与移动端要求

### 15.1 手机端优先

- 页面宽度从 360px 起可用。
- 底部导航常驻。
- 中间上传按钮突出。
- AI 对话按钮固定在搭配页右下角。
- 点击区域不小于 44px。
- 底部导航不遮挡保存按钮和 AI 按钮。
- 截图流采用双列布局。
- 衣柜页支持搜索。

### 15.2 桌面端适配

- 使用顶部导航。
- 内容区域设置最大宽度。
- 喜欢页可以从双列扩展为多列。
- 衣柜页可展示更多列。
- 搭配页可将老虎机和 AI 对话面板左右分栏。

## 16. 验收标准

### 16.1 上传

- 用户能通过中心 `+` 按钮选择拍照或图库。
- 上传成功后，单品出现在我的衣柜。
- 上传后的单品能进入对应品类老虎机。

### 16.2 AI 搭配

- 用户能在搭配页打开 AI 对话面板。
- 用户能输入文字需求。
- 系统能返回一套包含上衣、下装、鞋子的推荐。
- 三组老虎机能自动转动并停在推荐单品上。
- 缺少品类时有明确提示。

### 16.3 手动搭配

- 用户能手动滑动上衣、下装、鞋子。
- 当前选中三件单品能组成一套搭配。
- 用户能保存手动搭配。

### 16.4 我的喜欢

- 保存后能生成穿搭截图。
- 我的喜欢页以双列图片流展示截图。
- 点击截图能进入详情页。
- 详情页能展示大图和基础信息。

### 16.5 我的衣柜

- 能展示所有单品图片。
- 能按关键词搜索单品。
- 能按品类筛选。

### 16.6 社区

- `/community` 能展示 mock 社区内容。
- 不要求真实发布、评论、点赞入库。

## 19. 当前版本核心判断

第 3 版要把产品体验集中到一个更清楚的闭环：

```text
上传单品
-> 进入衣柜
-> AI 或手动搭配
-> 三组老虎机展示结果
-> 保存截图
-> 喜欢页双列展示
```

这比普通列表式衣柜更有辨识度，也更适合手机端。第一版先把这个闭环做顺，社区和语音可以保留入口，但不拖慢核心体验。
