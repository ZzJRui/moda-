# AI Closet Stylist MVP PRD

版本：v0.1  
日期：2026-07-04  
产品定位：Web 版 AI 数字衣柜 MVP

## 1. 一句话说明

用户上传真实单品图片，系统完成去背景、居中白底和标签识别；用户通过 AI 对话获得穿搭推荐，前端用老虎机式动画展示结果，用户可以保存整套穿搭截图到“我的喜欢”。

## 2. 背景与目标

很多穿搭工具把重点放在生成新衣服或真人试穿图上，第一版成本高、稳定性也不够好。本产品先解决更轻、更容易演示闭环的问题：把用户已有衣物数字化，并从这些衣物里推荐可保存的搭配。

MVP 的目标是跑通完整链路：

1. 上传真实衣物图片。
2. 生成白底单品图，并保存到数字衣柜。
3. 为单品生成基础标签。
4. 根据用户输入，从衣柜中选出上衣、下装和鞋子。
5. 用前端动画展示推荐过程。
6. 保存穿搭截图，并在收藏页查看。

## 3. 用户与使用场景

目标用户是希望快速管理衣柜、获得日常穿搭建议的个人用户。第一版按单用户本地演示设计，不做注册登录、多用户云同步和社交能力。

典型场景：

- 用户整理衣柜时，连续上传多件上衣、下装和鞋子。
- 用户出门前输入“今天想去公园玩，帮我搭配一套轻松一点的衣服”。
- 系统从已有衣物中选出一套搭配，并说明理由。
- 用户对结果满意后保存截图，以后可以在“我的喜欢”里回看。
- 用户不想问 AI 时，可以进入搭配页手动切换单品组合。

## 4. MVP 范围

### 4.1 本期做

- 数字衣柜页 `/closet`
- AI 对话页 `/chat`
- 我的喜欢页 `/favorites`
- 手动搭配页 `/style`
- 单品图片上传
- 原图和处理图本地存储
- SQLite 保存衣物、穿搭和收藏数据
- 基础图片处理：去背景、居中、白底
- 基础标签：品类、颜色、风格
- AI 或规则推荐穿搭
- 前端老虎机式推荐动画

### 4.2 本期不做

- 登录注册和多用户体系

## 5. 产品信息架构

### 5.1 数字衣柜页 `/closet`

页面目标：上传和管理单品。

核心能力：

- 用户上传一张衣物图片。
- 系统保存原图。
- 系统生成白底处理图。
- 系统识别或让用户确认品类、颜色和风格。
- 上传成功后，单品出现在衣柜列表中。

第一版支持的品类：

- 上衣：`top`
- 下装：`bottom`
- 鞋子：`shoes`

第一版支持的筛选：

- 全部
- 上衣
- 下装
- 鞋子
- 白色
- 黑色
- 休闲
- 甜酷
- 通勤
- 运动

必要状态：

- 上传中
- 处理失败
- 标签识别失败
- 衣柜为空
- 筛选无结果

### 5.2 AI 对话页 `/chat`

页面目标：让用户用自然语言获得穿搭推荐。

用户输入示例：

```text
今天想穿白色的衣服，帮我搭配一下。
```

```text
天气很好，我想去公园玩，帮我搭配一套衣服。
```

系统行为：

1. 前端提交用户输入。
2. 后端读取衣柜数据。
3. 推荐服务从用户已有单品中选择上衣、下装和鞋子。
4. 后端返回推荐单品 ID 和推荐理由。
5. 前端播放 1.5 到 3 秒动画。
6. 动画停在推荐单品上。
7. 用户可以保存当前穿搭截图。

推荐结果格式：

```json
{
  "top_id": 12,
  "bottom_id": 35,
  "shoes_id": 48,
  "reason": "白色上衣搭配浅色下装，整体更适合公园休闲场景，鞋子选择运动款可以提高舒适度。"
}
```

兜底规则：

- 如果某个品类为空，页面提示用户先补充该品类单品。
- 如果 AI 服务不可用，后端使用规则推荐：按用户关键词匹配颜色和风格，再随机补齐缺失品类。
- 如果推荐结果引用了不存在的单品 ID，后端重新选择有效单品。

### 5.3 我的喜欢页 `/favorites`

页面目标：查看已保存的穿搭截图。

保存逻辑：

1. 前端用 `html2canvas` 把当前穿搭区域转为图片。
2. 前端把图片上传到后端。
3. 后端保存截图文件，并记录对应穿搭数据。
4. 收藏页展示截图、推荐理由和保存时间。

说明：`html2canvas` 基于 DOM 渲染，不是系统级截图。第一版应避免过于复杂的滤镜、跨域图片和视频元素，减少截图偏差。

必要状态：

- 暂无收藏
- 保存中
- 保存失败
- 截图图片加载失败

### 5.4 手动搭配页 `/style`

页面目标：让用户不通过 AI，也能像玩衣柜一样浏览和组合衣物。

页面结构：

```text
上衣横向滑动区
下装横向滑动区
鞋子横向滑动区
```

核心能力：

- 每个品类单独横向切换。
- 当前选中的三件单品组成一套搭配。
- 用户可以保存当前组合截图。
- 当前品类为空时显示补充提示。

## 6. 核心流程

### 6.1 上传流程

```text
用户上传图片
-> FastAPI 接收文件
-> 保存原图到 uploads/original/
-> rembg 或第三方服务去背景
-> Pillow 居中并生成白底图
-> AI 或规则生成标签
-> 保存处理图到 uploads/processed/
-> 写入 SQLite
-> 返回单品数据给前端
```

上传限制：

- 支持 `jpg`、`jpeg`、`png`、`webp`
- 单张图片建议不超过 8 MB
- 后端需要校验 MIME 类型和文件扩展名
- 文件名使用 UUID，避免覆盖和路径注入

### 6.2 推荐流程

```text
用户输入穿搭需求
-> 前端请求 /api/chat/recommend
-> 后端读取所有可用衣物
-> 推荐服务选择 top、bottom、shoes
-> 后端返回推荐 ID 和理由
-> 前端播放老虎机式动画
-> 动画停在推荐单品
-> 用户保存截图
```

推荐原则：

- AI 不生成新衣服，只从用户已有衣物中选择。
- 结果必须引用真实存在的单品 ID。
- 推荐理由要解释颜色、风格或场景匹配，不写空泛评价。

## 8. API 设计

| 方法 | 路径 | 作用 |

| `POST` | `/api/clothes/upload` | 上传单品图片 |

## 9. 技术方案

| 层级 | 技术 |
| --- | --- |
| 前端 | React + Vite + TypeScript |
| UI | Tailwind CSS |
| 后端 | Python + FastAPI |
| 数据库 | SQLite |
| 图片存储 | 本地 `uploads/` |
| 去背景 | `rembg`，可替换为第三方 API |
| 图片处理 | Pillow |
| AI 标签 | 多模态模型；MVP 可先 mock 或规则兜底 |
| AI 推荐 | 文本或多模态模型；MVP 必须保证返回有效单品 ID |
| 截图保存 | `html2canvas` |
| 部署 | Nginx + FastAPI + 静态前端 |

SQLite 建议：

- 本地开发和 MVP 演示使用 SQLite。
- 上线演示可开启 WAL 模式。
- 如果后续加入多用户和高并发，再迁移到 PostgreSQL。

## 10. 前端交互与视觉要求

整体风格应像一个实用的衣柜工具，不做营销式首页。首屏直接进入产品操作界面。

基础要求：

- 导航清晰：衣柜、AI 对话、搭配、我的喜欢。
- 上传入口醒目，但不占满整个页面。
- 单品卡片包含图片、品类、颜色、风格。
- 动画结果区要稳定，不因图片尺寸变化导致跳动。
- 按钮、筛选、空状态和错误提示都要有明确反馈。

推荐动画：

- 三行分别代表上衣、下装、鞋子。
- 每行横向滚动，模拟老虎机效果。
- 最终停在推荐单品。
- 动画结束后显示推荐理由和保存按钮。
- 如果用户偏好减少动态效果，应允许关闭或缩短动画。

## 11. 验收标准

### 11.1 衣柜页

- 用户能成功上传支持格式的图片。
- 上传后能看到处理后的单品卡片。
- 单品至少有品类、颜色和风格。
- 按品类或标签筛选能正常工作。
- 上传失败时有明确提示。

### 11.2 AI 对话页

- 用户输入需求后能得到一套包含上衣、下装和鞋子的推荐。
- 推荐单品都来自衣柜已有数据。
- 衣柜缺少某个品类时，系统提示补充，不返回无效结果。
- 前端动画能播放并停在推荐项上。
- 推荐理由能解释搭配依据。

### 11.3 我的喜欢页

- 用户能保存当前穿搭截图。
- 收藏页能展示已保存截图。
- 刷新页面后收藏数据仍存在。

### 11.4 手动搭配页

- 用户能分别切换上衣、下装和鞋子。
- 当前组合能保存截图。
- 某品类为空时有补充提示。

## 12. 开发顺序

### 阶段一：静态前端

- 搭建 React + Vite + TypeScript 项目。
- 完成四个页面的静态 UI。
- 使用假数据展示单品、推荐和收藏。
- 完成推荐动画原型。

### 阶段二：后端和 SQLite

- 搭建 FastAPI 服务。
- 建立 SQLite 表。
- 实现上传、列表、穿搭和收藏 API。
- 暴露本地静态图片目录。

### 阶段三：图片处理

- 接入 Pillow。
- 接入 rembg 或临时跳过去背景。
- 生成居中白底图。
- 处理图片格式和尺寸。

### 阶段四：标签和推荐

- 先做 mock 或规则标签。
- 再接入多模态模型识别品类、颜色和风格。
- 推荐服务先按规则返回有效组合。
- AI 接入后保留规则兜底。

### 阶段五：截图保存

- 接入 `html2canvas`。
- 前端生成截图。
- 后端保存截图文件。
- 收藏页读取并展示。

## 13. 风险与修正建议

原方案方向清楚，没有需要暂停的问题，但有几处建议在执行时收紧：

- AI 能力不要作为第一天的硬依赖。先用 mock 和规则跑通闭环，再替换为真实模型。
- `rembg` 对环境和模型文件有依赖，部署前要验证冷启动、CPU 性能和图片处理耗时。
- 截图保存不能只停留在前端内存里，必须上传到后端，否则刷新后会丢失。
- 本地 `uploads/` 和 SQLite 适合 MVP，不适合多用户生产环境。
- 品类命名要统一使用 `shoes_id`，避免接口里同时出现 `shoe_id` 和 `shoes_id`。
- 需要限制上传大小和文件类型，避免异常图片拖垮服务。
- 如果前端图片跨域，`html2canvas` 可能截图失败。第一版应使用同源静态图片。

## 14. 暂不需要追问的假设

为了直接进入 PRD 阶段，本文先采用以下假设：

- 第一版面向单用户本地演示。
- 不做登录注册。
- AI 标签和 AI 推荐允许先用 mock 或规则替代。
- 图片统一存储在后端本地目录。
- 收藏保存的是穿搭截图，不是可编辑模板。
- 页面以桌面 Web 为主，同时保证基础移动端可用。

如果后续要进入开发计划，优先确认三件事：是否必须接真实 AI 模型、是否需要部署到公网、是否要求移动端拍照上传体验。
